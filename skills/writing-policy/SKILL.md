# 정책서 작성/갱신

기능 구현 완료 후 두 관점으로 문서화한다.

## 7-a. 기획 정책서

- **저장 위치:** `docs/policy/[기능명].md`
- **대상 독자:** 기획자, 운영자, 비개발자

**포함 내용:**
- 도입 목적
- 사용자에게 보이는 변화 (화면·동작 기준)
- 적용/제외 범위
- 관련 법령·약관 고지 필요 여부 및 후속 티켓 연결

**절대 포함하지 않는 항목:**
- 파일명·클래스명·함수명 (예: `MartCartAppBar`, `lib/mart/...`)
- 플레이버·빌드 타입 등 빌드 시스템 용어 → "커스텀 브랜드 버전" 등으로 대체
- 기술 스택명 (Flutter, Firebase, Dart 등)
- 코드 구조·아키텍처 설명
- API·데이터베이스 경로

> 초안 작성 후 자가 검토: **"기획자가 이 문서를 읽고 구현 없이 기능을 설명할 수 있는가?"**  
> 통과하면 제출.

## 7-b. 개발 독스

- **저장 위치:** `docs/dev/[기능명].md`
- **대상 독자:** 개발자

**포함 내용:**
- 연동 구조 및 초기화 흐름
- 환경변수 목록 및 주입 방식
- 핵심 파일 위치
- CI/CD 반영 여부

## 완료 기준

- [ ] 7-a, 7-b 모두 작성 또는 갱신 완료
- [ ] 약관·법령 고지 필요 시 운영 티켓 생성
